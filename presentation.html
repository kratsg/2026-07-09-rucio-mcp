Thanks for making time. Fifteen minutes: twelve of talking, three of live demo. One sentence up front: our users are already pointing LLM agents at Rucio, several of us have independently built MCP servers to make that safe, and I want to convince you that we should converge on one shared implementation rather than maintain three. I'll show you what rucio-mcp does, the architecture and security choices, and then we talk about the part Martin rightly cares about: who maintains it and how.

This is already happening, with or without us

  • LLM agents (Claude Code, Gemini CLI, Codex, …) are on our users' laptops and analysis facilities today
  • Physicists already ask them Rucio questions: where is my dataset, why is my rule stuck, do I have quota?
  • Without a proper interface, the agent improvises: pattern-matched CLI calls, scraped output, guessed conventions. And it makes those mistakes with your credentials
  • MCP (Model Context Protocol) is the industry standard for giving agents structured, permissioned access to a service

The question isn't whether agents talk to Rucio. It's whether they do it through an interface we design, review, and control.

Set the stakes without hype. The agents are already here; the unsafe path is the default path. An MCP server is how we make the interaction structured: typed tools, scoped permissions, the user's own identity. This framing matters for the security discussion later: the alternative to a reviewed MCP server is not "no agents", it's agents driving the CLI unsupervised.

Runs where your users are

stdio · local

Your laptop or login node

  • Uses your existing credentials: x509 proxy, userpass, or OIDC
  • Same environment as the rucio CLI: if that works, this works
  • One line in your MCP client config

http · multi-user

Hosted for a community

  • One deployment, many users, many sites (path-mounted per experiment)
  • OAuth 2.1 login in front → each user's own Rucio identity behind
  • No client software to install at all

http · shared secret

Pre-authenticated service

  • Single service identity (e.g. a VOMS proxy on a service host) behind a static bearer
  • For gated group accounts, dashboards, shifter bots

--read-only works in every mode: all write tools refuse before touching Rucio. Operators deploy with a Helm chart; Prometheus metrics built in.

Three deployment shapes cover the realistic uses: an individual physicist (stdio, their own creds), an experiment or facility hosting for everyone (http multi-user), and service or shifter automation (shared secret). Read-only mode is the nervous operator's friend and a sensible default for a first deployment. Helm chart and metrics because operators, not just users, have to love this.

Security was the design problem

An MCP server sits in a sensitive spot. That constraint shaped every architecture choice.

Directly acknowledge the concern Martin raised: this sits in a sensitive spot, and anything blessed by Rucio inherits Rucio's reputation. So instead of hand-waving, the next two slides show the two hardest design decisions and the resulting posture.

The OAuth bridge: agent-friendly auth, Rucio-native trust

the mismatch

  • MCP clients speak standard OAuth 2.1 (auth code + PKCE)
  • Rucio speaks its own OIDC polling flow; users never hold raw IdP tokens
  • Naive fixes: register every client at each IAM (unscalable) or proxy credentials (unacceptable)

the bridge

  • rucio-mcp presents an OAuth 2.1 authorization server to the client…
  • …and internally drives Rucio's own /auth/oidc polling flow
  • The MCP access token is the Rucio session token, passed through verbatim

  1. MCP client connects → browser opens → user logs in at their own IdP (CERN SSO, institutional login, …)

  2. Rucio's auth server mints the session token, exactly as it does for the CLI

  3. Every tool call then carries that token: the user's normal Rucio identity, permissions, and audit trail

No IAM registration by anyone: not operators, not users. User experience: click, log in, done.

This is the piece of engineering I'm proudest of, and it was built FOR Rucio, not adapted from a generic template. Rucio's OIDC flow was designed so users never hold IdP JWTs; MCP clients assume standard OAuth. The bridge satisfies both without weakening either: rucio-mcp drives the same polling flow the CLI uses, then hands the resulting Rucio session token to the MCP client verbatim. Full 12-step sequence diagram is in the docs and backup if anyone wants it in the discussion.

Security stance

  • Rucio stays the sole authority. Tokens are passed through, never wrapped or re-signed; a stale token is a 401 and a fresh login
  • No stored credentials. No secrets database; in-flight auth state is in-memory with a 5-minute TTL
  • Every action is the user's own identity. Normal Rucio ACLs, quotas, and audit logs apply unchanged
  • No open registration. Dynamic Client Registration removed; clients identified by CIMD metadata URLs, SSRF-guarded
  • Writes are opt-in. --read-only gates all mutating tools server-side
  • Observable. Prometheus metrics for auth outcomes, tool calls, and bridge sessions

Honest caveat: this layer is exactly what deserves community security review, and that's the argument for one codebase. One implementation means one review, hardened once, for everyone.

Walk the left column slowly; it's the trust story. The server deliberately holds no long-lived secrets and mints no tokens of its own, so compromising it yields little: Rucio still enforces every permission per-user. CIMD over DCR means there is no anonymous registration endpoint to abuse. And then the honest caveat, said out loud: I want this reviewed by people in this room. The pitch is not "trust me", it's "review one thing together instead of three things separately".

Last section before the demo: the actual ask. Two slides: why converging is worth it, and the governance/maintenance options on the table.

What converging buys us

security

One codebase to review, one auth layer to harden, one place to patch when the MCP spec or auth drafts move (they move fast: CIMD replaced DCR here within months)

effort

Rucio client quirks, pagination, error handling: fixed once. Experiments contribute the thin parts (a site preset and a nomenclature file), not whole servers

users

Same tools, same docs, same behavior whether you're on ATLAS, Belle II, or DUNE; support knowledge and prompts transfer between experiments

the prior art

Belle II and EIC's work isn't discarded: it validated the demand and shaped requirements. Missing a tool from those servers? That's an issue, not a fork. Ideas are very welcome upstream

This is how our community handles shared infrastructure everywhere else: we don't each maintain a private FTS. An agent interface deserves the same treatment.

Frame convergence as the normal move for this community, not a favor to me: we converged on Rucio itself, on FTS, on ROOT. Emphasize the welcoming part explicitly: if the Belle II or EIC servers do something rucio-mcp doesn't, I want that as an issue or PR and I'll happily co-credit; the tool inventory reached 44 partly by folding in exactly such requests. The AI space moves fast enough that duplicated maintenance is a real tax.

The honest slide: maintenance & governance

Rucio can't bless something unmaintained in a sensitive spot; agreed. Options, in increasing commitment:

option 1 · endorse

Community-blessed

Stays where it is; Rucio docs point to it as the MCP server. Lowest commitment, but the maintenance question is deferred, not answered

option 2 · affiliate

Rucio-adjacent project

Moves to a shared org with named co-maintainers from ≥2 institutes; Rucio team does a security review, no upstream burden

option 3 · adopt

Experimental upstream

Lives under rucio/ as an experimental component, inside Rucio's release & security process, with an agreed maintainer group

What I bring: continued active maintenance, releases, docs, and support. This is infrastructure I run in production, not a side demo.

What it needs from you: 1–2 co-maintainers, a security review pass, and a site owner per experiment.

This is Martin's question, taken seriously rather than dodged. Say plainly: today this has one active maintainer, and that is not sustainable for something Rucio blesses; I know that. The three options are a menu for the discussion after the demo, not a demand; option 2 is a realistic middle if institutes can name people. What I can honestly commit: I run this in production, so maintenance isn't charity for me. What I cannot do alone is be the bus-factor-one security surface for the whole community, which is precisely the argument for pooling into one implementation.

Backup: the bridge, end to end

  1. MCP client hits /authorize with a CIMD client_id (an https URL); server fetches + validates the metadata document

  2. rucio-mcp calls Rucio /auth/oidc, gets the user-facing polling URL, starts a background poll of /auth/oidc_redirect

  3. User logs in at their IdP in the browser; Rucio mints the session token exactly as for the CLI

  4. Poll completes → rucio-mcp mints a one-time auth code (PKCE-bound, 5-min TTL, in-memory)

  5. Client exchanges the code at /token → access_token is the Rucio session token; expires_in mirrors the token's real expiry, no refresh tokens

  6. Tool calls carry the bearer → injected as X-Rucio-Auth-Token; 401 from Rucio → client re-runs the flow

Full sequence diagram: rucio-mcp.readthedocs.io → “Rucio OAuth Bridge Architecture”. Upstream flow discussion: rucio/rucio#8568.

For auth-deep questions: why polling and not the webhome cookie (cross-domain fails) or fetchcode (manual copy-paste UX). Server-side polling is the one Rucio-native flow that works cross-domain with zero user friction.
